# vite ssr로 직접 SSR 프레임워크 구현하기

SSR 프레임워크를 사용할 때는 `서버에서 React를 그려준다`는 정도로만 이해하고 넘어가기 쉽다. 하지만 실제로는 현재 요청 URL을 해석하고, 그 URL에 필요한 데이터를 서버에서 먼저 모은 뒤, HTML을 만들고, 브라우저에서 다시 hydration 하는 일련의 과정이 있다. 이번 글에서는 Vite SSR과 React를 이용해 아주 작은 SSR 런타임을 직접 만들고, 여기에 loader 패턴, 동적 라우팅, SSR/CSR 혼합 렌더링, TanStack Router까지 적용하면서 SSR 프레임워크의 핵심 구조를 정리해본다.

## 왜 직접 구현해보려 했는가

Next.js, Remix 같은 프레임워크를 쓰면 SSR 자체는 어렵지 않게 적용할 수 있다. 문제는 프레임워크가 너무 많은 것을 대신 처리해주기 때문에, 실제로 어떤 시점에 데이터가 로드되고 어떤 기준으로 페이지별 데이터가 나뉘는지 체감하기 어렵다는 점이다.

직접 구현해보면 다음이 명확해진다.

- SSR 프레임워크의 본질은 `route -> loader -> render -> hydrate` 흐름이라는 점
- 컴포넌트 내부 `useEffect fetch`만으로는 SSR HTML에 데이터가 들어가지 않는다는 점
- 동적 페이지도 첫 요청에서는 서버가 먼저 렌더링한다는 점
- 프레임워크가 제공하는 규약이 왜 필요한지 이해할 수 있다는 점

## 가장 작은 SSR 구조 만들기

처음에는 가장 작은 형태의 SSR 구조부터 만들었다.

- `server.js`가 HTTP 요청을 받는다.
- `index.html` 템플릿을 읽는다.
- `entry-server`에서 React를 `renderToString` 한다.
- 렌더 결과를 `#root`에 주입한다.
- 브라우저는 이후 `entry-client`를 실행해서 hydration 한다.

이 단계에서 핵심은 SSR이 결국 "서버가 HTML 문자열을 만들어서 내려주는 것"이라는 점이다. 브라우저는 이미 완성된 마크업을 먼저 받고, 이후에 자바스크립트가 붙으면서 인터랙션이 가능해진다.

## 서버에서 데이터를 먼저 가져와 HTML에 주입하기

단순히 컴포넌트만 서버에서 렌더링하는 것으로는 실제 SSR의 장점이 잘 드러나지 않는다. 그래서 mock API를 하나 두고, 서버가 먼저 데이터를 조회한 뒤 그 데이터를 HTML에 포함시키는 방식으로 확장했다.

이 과정의 흐름은 다음과 같다.

1. 서버가 현재 요청을 받는다.
2. 서버가 API를 호출해 초기 데이터를 가져온다.
3. 그 데이터로 React를 서버에서 렌더링한다.
4. 같은 데이터를 `window.__INITIAL_DATA__`로 HTML에 주입한다.
5. 클라이언트는 hydration 시 이 데이터를 그대로 재사용한다.

이 구조를 통해 확인할 수 있었던 것은 `useEffect` 안에서 데이터를 가져오는 방식만으로는 페이지 소스에 데이터가 포함되지 않는다는 점이다. SSR HTML에 데이터를 넣으려면, 서버가 렌더 전에 먼저 데이터를 가져와야 한다.

## loader 패턴으로 데이터 로딩 분리하기

다음으로 정리한 것은 데이터 로딩 책임을 컴포넌트 밖으로 빼는 일이었다. 컴포넌트 내부에서 직접 fetch를 하지 않고, 페이지별로 필요한 데이터를 모으는 `loader`를 별도 계층으로 분리했다.

이렇게 분리한 이유는 명확하다.

- 서버와 클라이언트가 같은 데이터 로딩 로직을 공유할 수 있다.
- 컴포넌트는 데이터 조회가 아니라 렌더링에 집중할 수 있다.
- 여러 API를 병렬로 호출하는 구조를 만들기 쉽다.
- 특정 라우트에 어떤 데이터가 필요한지 명시적으로 관리할 수 있다.

여기서 중요한 점은 "프레임워크가 어떤 API를 자동으로 추론하는 것"이 아니라는 점이다. 결국 어떤 페이지에 어떤 데이터가 필요한지는 프론트엔드 개발자가 정해야 한다. 프레임워크는 그 규약을 일정한 방식으로 실행해주는 역할에 가깝다.

## API가 많아질 때의 구조

API가 하나일 때는 단순 fetch로도 버틸 수 있지만, 개수가 늘어나기 시작하면 금방 구조가 복잡해진다. 그래서 레이어를 나눴다.

- `api/*`: 개별 API 호출 함수
- `loaders/*`: 페이지 단위 혹은 도메인 단위 데이터 조합
- `pageLoader`: 현재 라우트에 맞는 loader 선택

홈 페이지를 예로 들면, `profile`, `notices`, `stats`를 각각 개별 API 함수로 분리하고, 홈 페이지 loader에서는 이 세 요청을 `Promise.all`로 병렬 실행해 데이터를 모으는 구조로 만들었다.

이 구조의 장점은 페이지에 필요한 데이터를 한곳에서 조합할 수 있다는 점이다. API 함수는 재사용 가능한 작은 단위로 유지하고, 페이지 loader는 그 API들을 조합하는 역할만 담당하게 된다.

## 동적 라우팅 직접 처리하기

그 다음에는 `/posts/:postId` 형태의 동적 라우팅 예제를 추가했다. 여기서 확인하고 싶었던 것은 동적 페이지도 SSR의 흐름이 똑같이 적용된다는 점이었다.

예를 들어 `/posts/42` 요청이 들어오면 서버는 다음 순서로 동작한다.

1. URL에서 `postId=42`를 추출한다.
2. 해당 포스트와 댓글 데이터를 서버에서 가져온다.
3. 그 데이터를 이용해 HTML을 서버에서 렌더링한다.
4. 브라우저에 완성된 HTML을 응답한다.

즉, 동적 페이지라고 해서 CSR이 되는 것이 아니다. 첫 요청은 여전히 서버가 렌더링하고, 클라이언트는 그 이후 hydration 만 수행한다.

## 한 페이지 안에서 SSR 컴포넌트와 CSR 컴포넌트 함께 쓰기

실무에서는 모든 UI를 SSR로만 처리하지 않는다. 어떤 영역은 SSR로 보내는 것이 맞고, 어떤 영역은 클라이언트에서 나중에 가져오는 편이 더 적절하다. 그래서 동적 포스트 페이지에서 SSR 영역과 CSR 영역을 의도적으로 섞어 구현했다.

- 포스트 본문과 댓글은 SSR
- reactions 위젯은 CSR

포스트 본문과 댓글은 첫 화면에 바로 보여야 하고, 검색 엔진이나 소셜 메타 관점에서도 의미가 있는 데이터다. 반면 reactions 같은 영역은 사용자 인터랙션 이후에 갱신되어도 문제 없고, 오히려 클라이언트에서 가져오는 쪽이 자연스럽다.

이 예제를 통해 정리한 기준은 아래와 같다.

- 첫 페인트에 꼭 필요하고 SEO에 영향이 있는 데이터는 SSR
- 실시간성, 개인화, 사용자 액션 중심의 데이터는 CSR

SSR과 CSR은 경쟁 관계가 아니라, 같은 페이지 안에서 역할이 다른 도구라고 보는 편이 맞다.

## TanStack Router로 라우팅 구조 정리하기

초기에는 직접 라우트 매칭 로직을 작성해서 현재 경로를 판별했다. 이후에는 TanStack Router를 적용해 라우트 정의, URL 파라미터 처리, 페이지 컴포넌트 구조를 좀 더 명확하게 정리했다.

여기서도 중요한 것은 라우터가 데이터 요구를 자동으로 알아내는 것이 아니라는 점이다. 라우터는 URL 구조와 컴포넌트 구조를 정리해주고, 어떤 라우트가 선택되었는지를 안정적으로 제공한다. 실제로 어떤 데이터를 서버에서 먼저 가져올지는 여전히 개발자가 loader 계층에서 정의해야 한다.

즉, TanStack Router를 붙인다고 해서 loader 개념이 사라지는 것이 아니라, 오히려 라우트와 데이터 로딩을 더 명시적으로 연결할 수 있게 된다.

## 지금 구조에서 화면이 그려지는 순서

현재 구조를 순서대로 정리하면 아래와 같다.

```text
브라우저 요청
  -> server.js
  -> 현재 URL에 맞는 loader 실행
  -> 필요한 API 병렬 호출
  -> entry-server에서 renderToString
  -> HTML + window.__INITIAL_DATA__ 응답
  -> 브라우저가 HTML을 먼저 그림
  -> entry-client 실행
  -> window.__INITIAL_DATA__로 hydrate
  -> CSR 컴포넌트가 필요한 추가 fetch 수행
```

이 흐름을 직접 구현해보면 SSR 프레임워크가 하는 일이 훨씬 선명하게 보인다. 프레임워크는 마법처럼 SSR을 만들어내는 것이 아니라, 결국 이 순서를 일관되게 수행할 수 있도록 규약과 도구를 제공하는 것이다.

## Next.js와 비교하면 무엇이 다른가

직접 구현한 구조에서 `loader`는 별도 함수로 분리되어 있다. 반면 Next.js App Router에서는 이런 loader 함수가 밖으로 드러나지 않고, 서버 컴포넌트와 프레임워크의 fetch 규약 안에 녹아 있다.

예를 들어 Remix 스타일은 "이 페이지의 데이터는 `loader()`에서 가져온다"가 명시적으로 보인다. 반면 Next.js App Router에서는 `async Page()` 컴포넌트 안에서 바로 데이터를 가져오는 경우가 많다. 이름이 다를 뿐, 현재 라우트에 필요한 데이터를 먼저 서버에서 가져온다는 점에서는 같은 역할을 한다.

즉 차이는 본질이 아니라 표현 방식에 있다.

- Remix/React Router 계열은 loader가 바깥으로 드러난다.
- Next.js App Router는 loader 역할이 서버 컴포넌트 안에 흡수되어 있다.

이 차이를 이해하고 나면 "SSR 프레임워크가 결국 무엇을 해주는가"를 더 잘 볼 수 있게 된다.

## 직접 구현해보며 이해한 SSR 프레임워크의 본질

이번 실험에서 가장 명확해진 것은 SSR 프레임워크의 핵심이 생각보다 단순한 계약이라는 점이었다.

- 현재 요청 URL을 해석한다.
- 해당 라우트에 필요한 데이터를 먼저 모은다.
- 그 데이터로 서버에서 HTML을 렌더링한다.
- 같은 데이터를 클라이언트에 전달해 hydration 한다.

프레임워크는 이 과정을 더 안전하고 일관되게 제공하는 도구에 가깝다. 직접 구현해보면 Next.js, Remix, Nuxt 같은 프레임워크가 왜 특정 규약을 강제하는지 이해가 쉬워진다. 결국 그 규약들은 모두 "현재 라우트에 필요한 데이터를 언제, 어디서, 어떤 방식으로 불러와서 HTML과 hydration 사이에 일관성을 유지할 것인가"라는 같은 문제를 풀기 위한 장치이기 때문이다.

직접 구현해본 뒤에는 SSR 프레임워크를 단순히 사용하는 입장이 아니라, 그 설계를 읽고 판단할 수 있는 시각이 생긴다. 그 점이 이번 작업에서 가장 큰 수확이었다.
